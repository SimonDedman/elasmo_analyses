# Interactive Validation UI — Design Spec

**Date:** 2026-04-16
**Author:** Simon Dedman + Claude
**Status:** Draft

## Problem

The current validation workflow uses an XLSX file with coauthor tabs (16 columns, ~700 rows per reviewer). This has three issues:

1. Shows what **was** triggered but not what **wasn't** — validators can't spot false negatives
2. Flat table format is hard to read — too many columns per row, no grouping
3. Free-text validation field produces unstructured prose instead of actionable, machine-readable feedback

The XLSX approach also doesn't scale beyond the current 10-person team. At Sharks International (May 2026), the project will be presented to the global elasmobranch research community — an opportunity to crowd-source validation from hundreds of domain experts during a narrow attention window.

## Solution

Static HTML validation pages hosted on GitHub Pages, one per author, with structured input controls and a serverless submission pipeline.

## Architecture

Three components:

### 1. Generation Script (`scripts/generate_validation_pages.py`)

Python script that reads the project's data files and produces static HTML.

**Inputs:**
- `outputs/literature_review_enriched.parquet` — 1,546 columns, 30,553 papers
- `outputs/schema_extraction_evidence.csv` — 231,753 evidence rows (audit trail for rule-based extractions)
- `outputs/openalex_paper_authors.csv` — 71,805 author-paper records
- `outputs/openalex_unique_authors.csv` — 28,952 unique authors with OpenAlex IDs

**Outputs:**
- `docs/validate/{openalex_id}.html` — one per author (or merged author group)
- `docs/validate/index.html` — landing page with name search
- `docs/validate/assets/` — shared CSS, JS
- `docs/404.html` — custom 404 directing to Shark References

**Author merge logic:** Group authors by `(display_name, most_common_institution)`. If multiple OpenAlex IDs share both fields, merge onto the primary ID (highest paper count). Secondary ID pages become HTTP redirects to the primary. The merged page notes which IDs were combined. Affects ~187 authors (0.65%).

### 2. Static Site on GitHub Pages

Served from `docs/` on the `main` branch of `SimonDedman/elasmo_analyses`.

- URL pattern: `simondedman.github.io/elasmo_analyses/validate/{openalex_id}.html`
- No Jekyll (`.nojekyll` already present in `docs/`)
- All interactivity is client-side JavaScript
- Validation state auto-saves to `localStorage`

### 3. Submission Pipeline

**Submit flow:**
1. Validator clicks "Submit" on their page
2. Browser POSTs JSON to GitHub API `repository_dispatch` endpoint
3. GitHub Action triggers: creates branch `validation/{openalex_id}/{timestamp}`, commits JSON to `validations/{openalex_id}_{timestamp}.json`, opens PR
4. PR body contains human-readable summary of changes (generated by the Action from the JSON diff)
5. PRs are reviewed — potentially semi-automated via Claude scheduled action for schema-valid, non-conflicting submissions

**Authentication:** `repository_dispatch` requires a GitHub token — it can't be called anonymously from a browser. Solution: a lightweight proxy that accepts unauthenticated POSTs and forwards to GitHub. Options in order of preference:
1. **Cloudflare Worker** (free tier, 100K requests/day) — holds the PAT as an environment secret, validates a shared token in the request payload, forwards to `repository_dispatch`
2. **Netlify/Vercel serverless function** — same pattern, different host
3. **Direct PAT in client** — least secure, only viable with a fine-grained token scoped to `actions:write` on this single repo; acceptable for a low-risk academic project but not ideal

The shared token in the payload (a simple string, not a PAT) prevents abuse — only requests with the correct token are forwarded.

## Page Layout

Each author's validation page contains:

### Header
- Author name, institution, OpenAlex ID
- Number of papers in database
- Merged-profile note (if applicable): "This page includes papers from OpenAlex profiles {id1}, {id2}"

### Per-Paper Block (collapsible, sorted by year descending)

Default collapsed. Summary line:

```
▶ Smith et al. (2019) — "Habitat use of reef sharks in..." — J. Fish Biology
```

Expanding reveals sections in this order:

#### Metadata Row (read-only)
Single row showing: year, journal, DOI (linked), OA status, Altmetric score, first author, study type, country, superregion, epoch. These are API lookups or metadata fields — not rule-editable.

#### Schema Categories (one row per category)

Each category shows **all options** as checkboxes — triggered ones pre-checked, untriggered ones unchecked but visible. Two tiers of UI based on whether extraction evidence exists:

**Tier 1 — Rule-based extractions (with evidence trail):**

| Schema | Prefix | Count | UI |
|--------|--------|-------|----|
| Ocean basin | `b_` | 9 | Checkbox grid |
| Discipline | `d_` | 19 | Checkbox grid |
| Ecosystem | `eco_` | 23 | Checkbox grid |
| Pressure | `pr_` | 26 | Checkbox grid |
| Impact | `imp_` | 25 | Checkbox grid |
| Gear | `gear_` | 29 | Checkbox grid |

Each triggered checkbox has an expandable details panel showing:
- Matched terms and frequencies
- Section where match occurred
- Threshold value
- Context snippet

Each category has:
- Traffic-light rating: Correct / Partially correct / Incorrect (radio buttons)
- The checkboxes themselves serve as the correction mechanism (uncheck false positives, check false negatives)
- Notes field (free text, for edge cases only)
- Rule feedback panel (optional, expandable):
  - Current threshold with +/- adjuster
  - "Suggest adding term" text input
  - "Suggest removing term" dropdown of current terms

**Tier 2 — Automated extractions (no evidence trail):**

| Schema | Prefix | Count | UI |
|--------|--------|-------|----|
| Species | `sp_` | 1,308 | Triggered species shown as checked tags; autocomplete search to add others |
| Analytical techniques | `a_` | 215 | Triggered techniques shown as checked tags; autocomplete search to add others |
| Ocean basin (geographic pipeline) | `ob_` | 9 | Checkbox grid — separate source from `b_`, both shown |
| Depth | `depth_` | 3 | Read-only display of range/min/max with editable override fields |

These show what was extracted and allow corrections, but have no matched-terms/threshold detail (different extraction mechanism).

**Future-proofing:** The generation script discovers schemas dynamically from column prefixes in the parquet. New schemas added to the extraction pipeline will automatically appear in validation pages on the next generation run. The UI tier (1 or 2) is determined by whether the prefix appears in the evidence CSV.

### Page Footer
- Auto-save indicator ("Last saved: 14:32")
- Progress bar: X of Y papers reviewed (a paper counts as "reviewed" when at least one category has a rating)
- "Submit validation" button
- "Download responses as JSON" fallback button

## Landing Page (`docs/validate/index.html`)

- Project title and brief description
- "Find your validation page" search box
- Client-side query to OpenAlex API: `https://api.openalex.org/authors?search={name}`
- Results show: name, institution, paper count, OpenAlex ID
- Clicking a result navigates to `/validate/{id}.html`
- Note explaining how to find your OpenAlex ID manually

## Custom 404 Page (`docs/404.html`)

For URLs under `/validate/`:
> "We don't have papers linked to this OpenAlex ID in our database. If your elasmobranch research is missing, please share your papers with [Shark References](https://shark-references.com) — they'll be included in our next monthly sync and extraction run."

For other 404s: generic message with link to the landing page.

The 404 page uses JavaScript to parse the attempted URL path — if it matches `/validate/{id}`, it displays the Shark References message. Otherwise it shows the generic 404.

## Pre-generation Strategy

For the Sharks International conference window:

1. **Pre-generate all 28,952 author pages** — estimated ~580 MB with shared template factored out. Within GitHub Pages 1 GB soft limit.
2. **Deterministic URLs** — anyone in the database can find their page immediately via the landing-page search or direct URL.
3. **404 fallback** for researchers not in the database directs to Shark References for paper submission.
4. **Re-generation** can be run incrementally as new papers are ingested or rules change.

## PR Automation

Each submitted validation creates a PR with:
- Branch: `validation/{openalex_id}/{timestamp}`
- File: `validations/{openalex_id}_{timestamp}.json`
- PR title: "Validation: {author_name} ({N} papers reviewed)"
- PR body: human-readable summary generated from JSON — lists each paper and what changed

Automation tiers:
1. **Immediate:** PR body summary makes manual review fast
2. **Near-term:** GitHub Action validates JSON schema, flags conflicts with other validators' submissions for the same papers
3. **Future:** Claude scheduled action auto-merges non-conflicting, schema-valid submissions and flags edge cases for manual review

## Dependencies

- Python: `pyarrow`, `jinja2` (HTML templating), standard library
- Client-side: vanilla JS (no framework), Tom Select (~15 KB, for species/technique autocomplete)
- GitHub: Pages enabled on `docs/`, repository secret for dispatch PAT
- OpenAlex API: free, no key needed (used client-side for name search on landing page)

## Out of Scope

- User authentication (validators are anonymous apart from their OpenAlex ID)
- Real-time collaboration (each validator works independently)
- Direct database writes (all changes go through PR review)
- Email notifications (deterministic URLs eliminate the need)
